## 1. 说明

按课程 Project 1-B 要求，本报告给出：

1. **对本组自己代码的测试结果**（根目录 `shannon_codes/`：`ShannonCode` / `HuffmanCode` 及字节流、`.shc` 封装层）；
2. **对 PK 对手代码的测试结果**（对手代码位于 `info_project_1B/`：`shannon.py` / `huffman.py`）；
3. **两份实现的客观分析与评价**（优势与不足）。

两份代码均**未作任何修改**，由 `pk_benchmark.py` 在同一进程中分别 import 调用，跑在**同一组中英文文本**上，统一口径采集指标；原始数据见 `pk_results.json`。

> 重点：因为双方实现的是同一套算法，**只要两边都正确，平均码长 L、效率 η、纯码流大小就完全一致**——这些是算法级量。PK 的真正胜负落在**正确性、文件封装、速度、工程健壮性**上。

## 4. 对本组自己代码的测试

- **Shannon**（`shannon.py` + `shannon_bytes.py` + `shannon_codec.py`）：累积概率 F 用浮点二进制展开**截断**取前 L 位（`_frac_to_binary`），是严格前缀码；`is_prefix_code()` 自检全部通过，两文本均**无损还原**。字节流版 `(code,length)` 联合键解码正确；`.shc` 单文件用 **canonical 码表（只存 token+count，二进制）** + 位流，解码端重算 Shannon 重建码表。
- **Huffman**（`huffman.py` + `huffman_cli.py`）：堆 + 虚节点构造 q∈[2,10] 的 q-ary 前缀码，q=2/3/4 全部无损、η＞97%，前缀自检通过。
- **结论**：本组 Shannon、Huffman 两套编解码在中英文上**全部正确无损**，指标贴合理论（Huffman η＞99%，Shannon 因次优构造 η 89%~95%）。

## 5. 对 PK 对手代码的测试

- **Huffman**（`info_project_1B/huffman.py`）：堆 + 虚节点 + CRC32 自包含 `.huff`，q=2/3/4 **全部无损**，指标与本组完全一致。封装完善。
- **Shannon**（`info_project_1B/shannon.py`）：**实测发现正确性缺陷**。其累积值用**向上取整** `v=(cum·2^L+total−1)//total`，破坏了前缀性：
  - 中文产生 **3 对**、英文产生 **4 对**前缀冲突（如英文 `'p'=110101` 是 `'y'=1101011` 的前缀）；
  - `shannon_decode` 在解自身输出时**抛出"无效比特流"异常**，直接运行 `python shannon.py` 会崩溃，连还原文件都生成不了。
  - **修正**：取整由 ceil 改为 **floor**（`v=(cum·2^L)//total`）即可——floor 截断使各码字落在互不重叠的二进制区间，前缀冲突归零，改后两文本均可无损还原（修正版 L/η 与本组一致）。

## 6. 客观对比与评价（优势与不足）

### 6.1 算法正确性 —— **本组胜**

| | Shannon 前缀码 | Shannon 无损 | Huffman 前缀码 | Huffman 无损 |
| --- | :---: | :---: | :---: | :---: |
| **本组** | ✅ | ✅ | ✅ | ✅ |
| 对手 | ❌（3/4 对冲突） | ❌ 崩溃 | ✅ | ✅ |

本组 Shannon 用浮点二进制展开（等价 floor 截断）天然满足前缀性并带自检；对手 Shannon 因 ceil 取整破坏前缀码、无法解码，是硬伤。Huffman 两边都对。

### 6.2 编码效率（L、η）—— **平手**

同一算法、同一文本，双方 L、η、纯码流逐位相同（见总表），互为印证：两边都正确实现了算法。Huffman 最优（η＞99%）、Shannon 次优（η 89%~95%），符合理论。

### 6.3 文件封装与压缩 —— **各擅胜场**

- **Shannon 封装：本组胜。** 本组 `.shc` 用二进制 canonical 码表，英文压到 **2580 B（80.3%，且可无损解码）**，中文 6054 B；对手把码表存成 `<U+XXXX>\t频率\t码字` 文本，膨胀严重（中文外置码表约 10.9 KB，合计 12464 B），且实际还解不开。
- **Huffman 封装：对手胜。** 对手 `.huff` 是头 + 序列化树 + CRC32 的紧凑自包含文件（中文 4988 B、英文 2225 B＝原文 69%）；本组 Huffman 为初版，只有 `.bits` + `.codebook.json`，JSON 冗余使中文容器达 35 KB——**这是本组明显不足**。

### 6.4 速度 —— **本组更快（但口径需说明）**

本组编码普遍在 0.1~0.2 ms，对手 Huffman 编码 2.5~5 ms、解码 2~4.5 ms。原因：对手 Huffman 每次都做树序列化 + CRC + 逐位字符串拼接；本组核心仅字典查表 + join。**但需声明**：本组 Huffman 未构建二进制容器（少做了打包/校验工作），速度对比并非完全对等；本组 Shannon 字节流/`.shc` 即使含打包仍更快。

### 6.5 工程健壮性与可扩展性

| 维度 | 本组 | 对手 |
| --- | --- | --- |
| 完整性校验 | 前缀码自检；`.shc` 有 MAGIC/版本号，但无 CRC | Huffman 有 **CRC32** 数据校验 ✅ |
| 自包含解码 | `.shc` 自包含；Huffman 需外带 JSON 码表 | Huffman `.huff` 完全自包含 ✅ |
| q 范围 | q ∈ [2,10] | q ∈ [2,16]，更宽 ✅ |
| 结构 | 分层（core/bytes/codec/tokens）+ dataclass + 指标/自检方法 ✅ | 函数式，面向文件，单文件直跑 |
| 对手致命点 | — | Shannon ceil 前缀缺陷、文本码表膨胀 ❌ |

### 6.6 改进建议

- **本组**：给 Huffman 补一个像 `.shc` / `.huff` 那样的紧凑二进制容器（存码长即可重建 canonical Huffman），替换臃肿的 JSON 码表；为 `.shc` 加 CRC；q 上限放宽到 16。
- **对手**：Shannon 取整改 floor 修复前缀缺陷，并把外置文本码表改为二进制紧凑存储、加完整性校验。

---

## 7. 结论

1. **正确性上本组占优**：本组 Shannon、Huffman 在中英文上全部无损；对手 Huffman 正确，但**对手 Shannon 因 ceil 取整破坏前缀码、无法解码**（改 floor 即修复）——这正体现了组间互测的价值。
2. **算法效率两边等价**：同算法同文本，L/η/纯码流逐位一致，Huffman（η＞99%）优于 Shannon（η 89%~95%），符合"Huffman 为最优前缀码"。
3. **封装各有所长**：Shannon 的紧凑容器本组更优（英文真正压缩到 80%），Huffman 的自包含容器对手更优（紧凑 + CRC），本组 Huffman 初版的 JSON 码表是主要短板。
4. **压缩效果强依赖字母表/文本长度比**：英文（小字母表）可压到 67%~80%，中文（大字母表、短文）受码表/树描述开销拖累，几乎压不动，此时码表的**二进制紧凑表示**最关键。
